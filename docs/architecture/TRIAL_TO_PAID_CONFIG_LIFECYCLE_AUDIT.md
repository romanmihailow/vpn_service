# Аудит: жизненный цикл подписки, WireGuard peer и VPN-конфига (trial → expiration → payment)

**Дата:** 2025-03-12  
**Контекст:** Пользовательский сценарий: trial/promo → конфиг отправлен → trial не использован → trial истёк → напоминания → оплата → пользователь продолжает использовать старый конфиг.  
**Гипотеза:** IP/peer переиспользуются или деактивируются так, что старый конфиг перестаёт работать, а путь после оплаты не защищён.

---

## 1. Executive summary

- **Краткий вывод:** Гипотеза **частично подтверждается**. После истечения trial IP освобождается в пул, peer удаляется из WireGuard — **старый конфиг технически перестаёт работать**. При оплате (YooKassa, баллы, промо «новая подписка») создаётся **новая** подписка с **новым** IP и **новыми** ключами, новый конфиг **всегда отправляется**. Основной риск — **не переиспользование IP**, а то, что пользователь **не замечает или не подменяет** новый конфиг и продолжает пытаться подключаться старым файлом.
- **Подтверждается ли гипотеза:** Частично. Переиспользование IP после trial есть (IP возвращается в пул и может быть выдан другому пользователю), но при оплате пользователь получает **новый** конфиг; старый уже не работает из-за удаления peer и освобождения IP.
- **После trial expiration:** Подписка деактивируется (`active=FALSE`), IP возвращается в пул, peer удаляется из WireGuard. Старый конфиг перестаёт работать.
- **После оплаты:** Создаётся новая запись подписки, новый IP из пула, новые WG-ключи, peer добавляется, конфиг отправляется в Telegram.
- **Главный риск:** Пользователь не открывает/не импортирует новый конфиг после оплаты и продолжает использовать старый файл, который уже не может установить туннель (peer удалён, IP мог быть выдан другому).

---

## 2. Карта жизненного цикла

Пошагово по коду:

| Шаг | Событие | Участники (файл / функция) | БД / WG |
|-----|--------|-----------------------------|---------|
| **A** | Новый пользователь получает trial / promo | `tg_bot_runner`: `try_give_referral_trial_7d`, промо (promo apply → новая подписка) | `insert_subscription`, IP из пула, peer в WG |
| **B** | Бот выдаёт конфиг | `bot.send_vpn_config_to_user` | Конфиг строится из `wg.build_client_config( client_private_key, client_ip )` |
| **C** | Trial заканчивается (expires_at) | Job `auto_deactivate_expired_subscriptions` (каждые 60 с) | `get_expired_active_subscriptions` → `deactivate_subscription_by_id` |
| **D** | После окончания срока | `db.deactivate_subscription_by_id(sub_id, "auto_expire", release_ip_to_pool=True)` | `active=FALSE`, при `release_ip_to_pool=True` → `release_ip_in_pool(vpn_ip)` |
| **E** | Peer / IP после expire | Тот же job после деактивации | `wg.remove_peer(pub_key)`; IP в пуле `allocated=FALSE` |
| **F** | Пользователь оплачивает (YooKassa) | `yookassa_webhook_runner.process_yookassa_event` → payment.succeeded | `get_active_subscriptions_for_telegram` (только active и expires_at>NOW); для первого платежа после trial — пусто |
| **G** | Создание paid subscription | YooKassa: при `base_sub is None` | `deactivate_existing_active_subscriptions` (no-op), новый ключ, `wg.generate_client_ip()` → `allocate_free_ip_from_pool`, `wg.add_peer`, `insert_subscription`, `send_vpn_config_to_user` |
| **H** | Какой конфиг получает пользователь | Всегда конфиг **текущей** подписки (новой): новый IP, новые ключи. | Одна запись в `vpn_subscriptions` с новым `vpn_ip`, `wg_private_key`, `wg_public_key` |

Цепочка: **trial issued → config sent → expired → cleanup (deactivate + release IP + remove peer) → payment → new subscription (new IP, new keys) → new config sent.**

## 3. Назначение IP

- **Где назначается:**  
  - `app/wg.py`: `generate_client_ip()` → `db.allocate_free_ip_from_pool()`.  
  - Вызовы: выдача trial (`try_give_referral_trial_7d`), оплата баллами (новый конфиг), промо (новая подписка), YooKassa/Heleket при создании новой подписки, ручное добавление админом.

- **Алгоритм выбора IP:**  
  В `db.allocate_free_ip_from_pool()`: из таблицы `vpn_ip_pool` выбирается одна строка с `allocated = FALSE`, блокируется `SELECT ... FOR UPDATE SKIP LOCKED`, помечается `allocated = TRUE`, `allocated_at = NOW()`. Пул заполняется отдельно (миграции/скрипты); в коде приложения нет «следующего свободного октета» для выдачи — только пул.

- **Освобождается ли IP после окончания подписки:**  
  **Да.** В `db.deactivate_subscription_by_id(..., release_ip_to_pool=True)` (значение по умолчанию): если у данного IP нет других активных подписок (`active=TRUE`), вызывается `release_ip_in_pool(vpn_ip)` → в `vpn_ip_pool` выставляется `allocated = FALSE`, `allocated_at = NULL`.

- **Когда именно освобождается:**  
  - При **авто-истечении**: job `auto_deactivate_expired_subscriptions` вызывает `deactivate_subscription_by_id(sub_id, "auto_expire")` без третьего аргумента → `release_ip_to_pool=True` → IP освобождается сразу при деактивации.  
  - При **ручной деактивации** (`/admin_deactivate`): то же поведение.  
  - При **refund/cancel** в YooKassa: деактивация подписки с тем же механизмом.  
  - **Не** освобождается только когда явно передаётся `release_ip_to_pool=False` (например, при оплате баллами/промо с переиспользованием той же подписки — см. ниже).

- **Может ли IP быть выдан другому пользователю:**  
  **Да.** После `release_ip_in_pool` IP снова доступен в пуле и может быть выдан любому следующему запросу `allocate_free_ip_from_pool()`.

- **Reserved IP / grace period / cooldown:**  
  В текущей реализации **нет**: после деактивации подписки IP сразу возвращается в пул и может быть выдан другому пользователю. Поля `allocated_at` в пуле не используются для задержки переиспользования.

## 4. Поведение WireGuard peer и ключей

- **Удаляется ли peer после окончания trial / подписки:**  
  **Да.** В job `auto_deactivate_expired_subscriptions` после `deactivate_subscription_by_id` вызывается `wg.remove_peer(pub_key)`. Peer удаляется из интерфейса WireGuard и из конфига `wg0.conf` (блок с `# auto-added by vpn_service` и данным PublicKey).

- **Когда именно удаляется:**  
  В том же цикле, что и деактивация подписки: сразу после успешной `deactivate_subscription_by_id` для каждой истёкшей подписки.

- **Остаются ли у старой подписки ключи и IP в БД:**  
  В записи `vpn_subscriptions` поля `wg_public_key`, `wg_private_key`, `vpn_ip` **не очищаются** — подписка просто переводится в `active=FALSE`. Но peer в WG уже удалён, IP в пуле помечен как свободный, поэтому старый конфиг по этим данным **не может установить туннель**.

- **При новой оплате (после истёкшего trial):**  
  - Создаётся **новая** подписка (`insert_subscription`).  
  - Используются **новые** WG-ключи (`wg.generate_keypair()`).  
  - Выделяется **новый** IP из пула (`wg.generate_client_ip()` → `allocate_free_ip_from_pool`).  
  - В WG **добавляется новый** peer (`wg.add_peer`).  
  - Старый peer уже удалён на шаге expire; старые ключи в новой подписке не используются.

- **Может ли пользователь продолжать работать со старым конфигом после оплаты:**  
  **Нет.** Старый конфиг привязан к старому public key и старому IP. Peer с этим public key удалён из WireGuard, старый IP мог быть выдан другому. Технически старый конфиг **обязан** перестать работать: сервер больше не принимает этот peer, и адрес мог быть переприсвоен.

- **Переиспользование ключей/IP:**  
  Есть только в сценариях, когда **ещё есть действующая** подписка с теми же ключами:  
  - Оплата **баллами**: если `get_latest_subscription_for_telegram` возвращает подписку (active и expires_at > NOW) с ключами и IP — переиспользуются ключи и IP, peer не удалялся, новый конфиг не отправляется.  
  - **Промо** (новая подписка): если `get_latest_subscription_for_telegram` возвращает подписку с ключами и IP — переиспользуются ключи и IP, конфиг не переотправляется.  
  После **истечения** trial `get_latest_subscription_for_telegram` возвращает `None` (условие `active=TRUE AND expires_at > NOW()`), поэтому при первой оплате после trial переиспользования нет — всегда новая подписка, новый конфиг.

## 5. Flow trial expired → paid subscription

- **Создаётся новая запись подписки или продлевается старая:**  
  При первом платеже после истёкшего trial **всегда создаётся новая** запись.  
  - **YooKassa:** `get_active_subscriptions_for_telegram` возвращает только подписки с `active=TRUE` и `expires_at > NOW()`. Истёкший trial уже деактивирован → список пуст → `yookassa_sub = None` → `base_sub = None` → ветка «нет активной подписки — создаём новую»: `insert_subscription`, новый IP, новые ключи, `send_vpn_config_to_user`.  
  - **Оплата баллами:** `get_latest_subscription_for_telegram` тоже требует `active=TRUE` и `expires_at > NOW()` → после истечения trial возвращается `None` → нет переиспользования → новая подписка, новый конфиг отправляется.  
  - **Промо (новая подписка, ошибка «expired_or_inactive»):** та же логика с `get_latest_subscription_for_telegram` → при истёкшем trial — новая подписка, новый конфиг.

- **Какая подписка считается «latest»:**  
  `get_latest_subscription_for_telegram`: одна подписка с `telegram_user_id = %s` и `active = TRUE` и `expires_at > NOW()`, сортировка `ORDER BY id DESC`, `LIMIT 1`. То есть последняя по `id` среди **действующих**.

- **Какой конфиг должен использовать пользователь:**  
  После оплаты — **только** конфиг новой подписки (новый файл/QR из сообщения бота). Старый trial-конфиг невалиден (peer удалён, IP освобождён).

- **Может ли старый конфиг работать:**  
  **Нет.** Peer удалён в `auto_deactivate_expired_subscriptions`, IP возвращён в пул. Старый конфиг не может установить handshake.

- **Отправляется ли новый конфиг автоматически после оплаты:**  
  **Да.** Во всех сценариях создания новой подписки после оплаты (YooKassa, баллы с новым конфигом, промо с новой подпиской) вызывается `send_vpn_config_to_user` с конфигом, собранным из новых ключей и нового IP.

- **Может ли пользователь остаться со старым trial-конфигом:**  
  Не по технической ошибке системы: новый конфиг отправляется. Риск только в том, что пользователь **не замечает** новое сообщение с конфигом или не подменяет старый файл в клиенте на новый.

## 6. Resend / latest subscription audit

- **Как работает `get_latest_subscription_for_telegram`:**  
  Запрос: `WHERE telegram_user_id = %s AND active = TRUE AND expires_at > NOW() ORDER BY id DESC LIMIT 1`. Возвращается одна последняя по `id` **активная и не истёкшая** подписка.

- **Риск взять не ту подписку:**  
  При одном пользователе и одной активной подписке — риска нет. Если бы у пользователя было несколько активных подписок (в текущей логике перед созданием новой все активные деактивируются), то «latest» — максимальный `id`, то есть последняя созданная. Resend и отправка конфига после оплаты всегда привязаны к этой одной активной подписке. Сценария «отправить конфиг от другой подписки» при текущей схеме не возникает.

- **Ситуация «несколько подписок, бот отправляет не от той»:**  
  Не реализована: у пользователя в один момент времени после деактиваций остаётся не более одной активной подписки с `expires_at > NOW()`. Resend и AI `action_resend_config` используют `get_latest_subscription_for_telegram` → одна подписка.

- **После оплаты: «latest» одна, пользователь открывает файл от другой:**  
  После первой оплаты после trial у пользователя одна активная подписка — та, что только что создана. Других активных нет. «Открыть файл от другой» возможно только если пользователь вручную хранит старый файл (trial) и использует его — но этот файл уже нерабочий (peer удалён).

## 7. Cleanup / expire jobs

| Функция | Файл | Что делает | Влияет ли на старый конфиг |
|--------|------|------------|----------------------------|
| `auto_deactivate_expired_subscriptions` | `tg_bot_runner.py` | Раз в 60 с получает `get_expired_active_subscriptions()` (active=TRUE, expires_at<=NOW()), для каждой вызывает `deactivate_subscription_by_id(sub_id, "auto_expire")` и `wg.remove_peer(pub_key)`. IP возвращается в пул внутри `deactivate_subscription_by_id`. | Да: деактивация + удаление peer + освобождение IP делают старый конфиг нерабочим. |
| `auto_notify_expiring_subscriptions` | `tg_bot_runner.py` | Уведомления за 3 дня, 1 день и ~1 час до окончания срока; не меняет подписку, не удаляет peer, не освобождает IP. | Нет. |
| `deactivate_existing_active_subscriptions` | `tg_bot_runner.py` | Вызывается перед выдачей нового доступа (trial, оплата баллами, промо, YooKassa при новой подписке). Деактивирует все активные подписки пользователя, при `release_ips_to_pool=True` освобождает IP, удаляет peer. | Да, если вызывается с release_ips_to_pool=True (по умолчанию). |
| `deactivate_subscription_by_id` | `db.py` | Ставит подписке `active=FALSE`, при `release_ip_to_pool=True` и отсутствии других активных подписок на этот IP вызывает `release_ip_in_pool`. Peer не трогает (удаление делают вызывающие). | Да: освобождение IP. |
| Refund / cancel в YooKassa | `yookassa_webhook_runner.py` | Деактивация подписки по событию возврата/отмены, вызов `wg.remove_peer`. | Да. |

Отдельных «cleanup» задач, которые бы только освобождали IP без деактивации или удаления peer, в коде нет. Освобождение IP привязано к деактивации подписки.

## 8. Feasibility of 30-day IP reservation (grace period)

- **Возможно ли на текущей архитектуре:**  
  Да. Пул хранится в `vpn_ip_pool` (ip, allocated, allocated_at). Сейчас задержки переиспользования нет: при деактивации подписки сразу вызывается `release_ip_in_pool`. Чтобы держать IP за пользователем N дней, нужно **не** вызывать `release_ip_in_pool` при истечении подписки, а либо:  
  - ввести отдельное состояние «зарезервирован до даты X» и не отдавать такой IP в `allocate_free_ip_from_pool` до истечения X, либо  
  - не помечать IP как свободный при деактивации, а помечать «освободить после даты X» и учитывать это в логике выделения.

- **Что менять:**  
  - **Логика выделения IP:** в `allocate_free_ip_from_pool` (или аналоге) не выдавать IP, которые в «grace» периоде (например, по новой таблице или полю `released_at` / `reuse_after`).  
  - **Логика cleanup:** при expire не вызывать `release_ip_in_pool` сразу, а записывать дату, после которой IP можно переиспользовать (или переносить в отдельную таблицу «временно закреплённых»).  
  - **Логика выбора подписки при оплате:** при оплате в течение N дней после истечения trial можно искать **последнюю подписку пользователя по id** (включая неактивные) и при наличии у неё ключей и IP решать: продлевать её (реактивировать + не трогать peer, если peer ещё не удалён — но сейчас peer уже удалён при expire) или выдавать тот же IP заново. Сейчас при expire peer уже удалён, поэтому «вернуть тот же конфиг» без повторной выдачи того же IP и повторного добавления peer невозможно; вариант — не удалять peer при expire, а только деактивировать подписку и отложить освобождение IP на N дней (тогда при оплате в течение N дней можно было бы реактивировать подписку и добавить peer обратно, если конфиг тот же).  
  - **Риски:** исчерпание пула (меньше свободных IP на период grace), «залипшие» адреса у невернувшихся пользователей, необходимость админки/мониторинга по таким IP, возможный рассинхрон между записью подписки (vpn_ip остаётся у старой записи) и реальным состоянием peer в WG (если peer удалять при expire, то при продлении нужна повторная регистрация peer с тем же ключом и IP).

- **Итог:** Реализуемо за счёт отложенного освобождения IP и (при желании «тот же конфиг») — либо не удалять peer при expire и только помечать подписку неактивной с «reuse_after», либо при оплате в окне N дней искать последнюю подписку пользователя (включая неактивные), переиспользовать её IP/ключи и заново добавлять peer. Текущий код такой логики не содержит.

## 9. Final conclusion

- **Основной корень проблемы:**  
  Не переиспользование IP как таковое (новый конфиг после оплаты всегда отправляется и привязан к новой подписке), а то, что **пользователь не заменяет старый конфиг на новый**: не замечает сообщение с конфигом после оплаты или продолжает использовать старый файл. Старый конфиг при этом уже не работает (peer удалён, IP освобождён).

- **Вторичный фактор:**  
  Явного «обязательного» повторного шага «импортируй новый конфиг» после оплаты в UX может не хватать (одно сообщение с файлом легко потерять среди других уведомлений).

- **Минимальные пути исправления (без реализации в коде):**  
  1. **UX:** После успешной оплаты (и отправки конфига) явно показывать короткое напоминание: «Используй конфиг из этого сообщения; старый файл больше не подойдёт» и/или дублировать напоминание через N часов с кнопкой «Получить конфиг» (resend).  
  2. **Документация / поддержка:** В справке и ответах поддержки явно указывать: после смены подписки (например, окончание trial и новая оплата) нужно использовать новый конфиг из последнего сообщения бота.  
  3. **Опционально — grace period:** Ввести отложенное освобождение IP (например, 30 дней) и при оплате в этот период по возможности переиспользовать тот же IP (и при желании те же ключи) и явно сообщать пользователю: «Твой прежний конфиг снова активен» или «Используй тот же конфиг» — чтобы снизить путаницу. Это требует доработки логики expire/cleanup и выделения IP, как в п. 8.

## Ответы на итоговые вопросы

- **Подтверждается ли гипотеза про IP reuse:**  
  IP после trial действительно освобождается и может быть выдан другому. Но при оплате пользователь получает **новый** IP и новый конфиг; проблема не в том, что «ему выдали чужой старый конфиг», а в том, что он продолжает использовать **свой** старый trial-конфиг, который уже невалиден.

- **Отправляется ли новый конфиг после оплаты:**  
  **Да**, во всех сценариях первой оплаты после истёкшего trial (YooKassa, баллы, промо-новая подписка).

- **Может ли пользователь продолжать использовать старый trial-конфиг:**  
  **Нет**, технически он не работает: peer удалён, IP освобождён. Пользователь может только **пытаться** его использовать, и подключение не установится.

- **Реалистичен ли 30-day grace period:**  
  **Да**, за счёт отложенного освобождения IP и при необходимости доработки логики при оплате (переиспользование последней подписки пользователя в окне N дней).

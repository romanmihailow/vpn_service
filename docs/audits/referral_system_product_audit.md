# Referral System Audit Report

Продуктовый и технический аудит реферальной системы после последних изменений (одно уведомление о баллах с кнопками, удаление дублирующего короткого сообщения).

---

## 1. Funnel integrity

**OK**

Цепочка проверена по коду:

- **Реферал:** связь создаётся в `register_referral_start` при `/start` с реферальным кодом; запись в `referrals` (referrer ↔ referred).
- **Доступ:** приведённый получает подписку (триал или оплата) через обычные сценарии (YooKassa / Heleket).
- **Handshake:** фиксируется в `get_subscriptions_for_new_handshake_admin` по данным WireGuard; для подписки без записи `handshake_user_connected` отправляется сообщение пользователю и создаётся запись; затем при наличии реферера и включённых уведомлений — уведомление рефереру (`referral_user_connected`) и одна запись на подписку.
- **Оплата:** webhook вызывает `apply_referral_rewards_for_subscription` (начисление баллов по цепочке рефереров), затем в цикле по `awards` — отправка `send_referral_reward_notification` каждому рефереру и при необходимости одна запись `referral_points_awarded` на подписку.

Разрывов по этапам нет: от входа по ссылке до handshake и оплаты все события опираются на одни и те же сущности (referrals, subscriptions, subscription_notifications). Единственная потенциальная «дыра» — пользователь может оплатить без handshake (оплата до первого подключения); реферер тогда получит только уведомление о баллах, без «подключился». Это не баг, а особенность порядка событий.

---

## 2. Trigger correctness

**OK** (с оговоркой по сбоям отправки)

### referral_user_connected

- Отправляется только в job `auto_new_handshake_admin_notification`: после успешной отправки пользователю сообщения `handshake_user_connected` и при условиях: есть `referrer_id` (`get_referrer_telegram_id`), у реферера включены уведомления (`is_ref_connected_notification_enabled`), для подписки ещё нет записи `referral_user_connected`.
- Дублей нет: защита `has_subscription_notification(sub_id, "referral_user_connected")`; запись создаётся сразу после отправки.
- Пропуск возможен только если отправка в Telegram падает (exception): тогда реферер не получает сообщение, запись не создаётся, повторной отправки нет.

### referral_points_awarded (сообщение и запись)

- Сообщение о баллах отправляется один раз каждому рефереру в цепочке при каждой оплате (в цикле по `awards` после `apply_referral_rewards_for_subscription`). Запись `referral_points_awarded` в CRM создаётся один раз на подписку (первый проход по циклу, где `!has` и `is_ref_points_notification_enabled`).
- «Начислили, но не уведомили»: баллы начисляются в `apply_referral_rewards_for_subscription` до цикла отправки. Если отправка падает, пользователи могут не получить сообщение, но баллы уже в БД — сценарий возможен при сбое Telegram/сети.
- «Уведомили, но не начислили»: невозможно, начисление всегда перед отправкой.
- Двойная отправка при повторном webhook исключена: YooKassa и Heleket используют идемпотентность по `subscription_exists_by_event` (event = payment_id/uuid), повторная обработка того же платежа не выполняется.

## 3. UX quality

**OK** (система понятна, есть запас по «теплоте» и мотивации)

- **referral_user_connected:** короткий текст «🔥 Есть результат! Пользователь по вашей ссылке подключил VPN 👍», одна кнопка «🤝 Пригласить друга». Причина и результат ясны, действие очевидно; перегруза нет.
- **Уведомление о баллах:** заголовок «🎁 Тебе начислены реферальные баллы!», причина («по твоей реферальной цепочке»), сумма и уровень, тариф, две кнопки «🎮 Оплатить баллами» и «🤝 Пригласить друга». Структура «причина — результат — действие» выдержана.
- Сообщения не перегружены, без техжаргона. Резерв по продукту: можно усилить мотивацию (например, «ещё X баллов — скидка», прогресс, примеры выгоды), но база для понимания и следующих шагов есть.

## 4. Conversion points

**OK**

- После «подключился»: одна кнопка «Пригласить друга» → тот же referral flow (ссылка, шаринг).
- После «начислены баллы»: две кнопки — «Оплатить баллами» (переход в оплату баллами) и «Пригласить друга» (тот же flow). Оба callback обрабатываются, tracking пишется.
- Потеря пользователя «после уведомления» возможна только если он не нажимает кнопки; сам по себе следующий шаг не скрыт и не сломан.

---

## 5. Callback flows

**OK**

- `points:open:from_referral:{id}`: обрабатывается в `points_open_from_referral_callback`; парсится `referred_sub_id`, при отсутствии записи создаётся `referral_points_awarded_pay_clicked`, затем показ того же flow, что и для обычного `points:open` (баланс, тарифы, оплата баллами).
- `ref:open_from_referral:points:{id}`: в `ref_open_from_referral_callback` для контекста `points` записывается `referral_points_awarded_ref_clicked`, затем тот же referral flow (ссылка).
- Старые сценарии (`ref:open_from_notify`, `points:open` без суффикса) не подменяются: фильтр по `F.data.startswith("points:open:from_referral:")` и разбор `ref:open_from_referral:*` по контексту не трогают остальные callback.
- Tracking записывается по `referred_sub_id`, дедупликация через `has_subscription_notification`; некорректный/битый `id` обрабатывается (ответ пользователю без падения).

## 6. CRM consistency

**OK**

- Все реферальные типы учтены в `get_crm_funnel_report`: `referral_user_connected`, `referral_user_connected_ref_clicked`, `referral_points_awarded`, `referral_points_awarded_ref_clicked`, `referral_points_awarded_pay_clicked`; подсчёт по `COUNT(DISTINCT subscription_id) FILTER (WHERE notification_type = ...)`.
- Двойные записи по одному и тому же событию для одной подписки исключены: создание записей только после успешной отправки и под условием `!has_subscription_notification(..., type)`.
- Пропуск записи возможен только при сбое отправки (тогда сообщение не ушло — запись не создаём, CRM не завышен).
- Семантика: `referral_points_awarded` = число подписок (referred), по которым было создано хотя бы одно уведомление о баллах (одна запись на подписку при мультиуровне); клики привязаны к тем же подпискам. Рассинхрона между отправкой и отчётом нет.

## 7. Edge cases

**OK** (поведение определено; есть известные ограничения)

- **Пользователь без реферера:** `get_referrer_telegram_id` → `None`, уведомления рефереру не отправляются; при оплате `upline` пустой, начислений и сообщений о баллах нет. Ожидаемо.
- **Реферер с выключенными уведомлениями:** для «подключился» проверяется `is_ref_connected_notification_enabled` — сообщение не отправляется, запись не создаётся. Для «баллы» отправка сообщения идёт всем реферерам в цепочке (как раньше длинное сообщение), а запись `referral_points_awarded` создаётся только при включённой настройке — возможна небольшая асимметрия (получил сообщение, но в CRM по этой подписке не учтён), по дизайну.
- **Ошибка отправки в Telegram:** для `referral_user_connected` и для цикла баллов исключение логируется, запись не создаётся; повторной отправки нет. Возможна потеря одного уведомления при временном сбое.
- **Повторные webhook:** идемпотентность по `payment_id`/uuid через `subscription_exists_by_event` в YooKassa и Heleket; повторная обработка того же платежа не выполняется, двойного начисления и двойных рассылок нет.

---

## 8. Production risks

- **Потеря одного уведомления при сбое Telegram:** при падении `send_message` реферер не получает «подключился» или одно из сообщений о баллах; повторной отправки нет. Риск ограниченный, но присутствует.
- **Частичный сбой в цикле баллов:** если отправка падает на втором/третьем реферере в цепочке, предыдущие уже получили сообщение, последующие — нет; запись `referral_points_awarded` может быть создана после первого успешного send. Поведение предсказуемо, но не идеально при нестабильной сети.
- **Дубли:** при текущей идемпотентности webhook и проверках `has_subscription_notification` дублирования отправок/записей не ожидается.
- **Race conditions:** один payment_id обрабатывается одним инстансом webhook; handshake job обрабатывает подписки по данным WireGuard с проверкой «ещё не отправляли» — при нормальной задержке дублей не должно быть. Теоретически при очень быстром двойном прогоне job для одной подписки возможна гонка до записи `handshake_user_connected`/`referral_user_connected`; на практике маловероятно.
- **Silent-ошибки:** исключения в отправке и создании записей логируются (warning/error), не глотаются полностью; «молчаливых» падений без следа в логах не видно.

---

## 9. Growth potential

**Оценка: MEDIUM**

- **Есть:** начисление баллов и отображение суммы в уведомлении; уровень реферала; явные CTA («Оплатить баллами», «Пригласить друга»); ежедневный дайджест по сети 2+ уровня; настройки уведомлений.
- **Нет в текущей реализации:** явного «прогресса» (счётчик приглашённых, прогресс-бар до следующей награды), геймификации (уровни, бейджи), лидербордов, подсказок вида «ещё N баллов — скидка» или «приведи ещё одного — получи X».
- Система в первую очередь **информирует и ведёт к действию** (поделиться ссылкой, потратить баллы). Для роста виральности и удержания есть запас: добавить счётчики, прогресс и простую геймификацию поверх текущей базы.

---

## 10. Final verdict

**STABLE**

- Воронка цельная, триггеры корректны, дубли устранены, callback и CRM согласованы с реальными событиями.
- Edge cases обработаны; оставшиеся риски — сбои доставки и частичные отказы в цикле рассылки, без изменения логики они допустимы.
- UX достаточен для понимания и конверсии; рост потенциала — средний, улучшения возможны за счёт прогресса и мотивации, а не за счёт исправления ошибок.

Рекомендация: выкатывать в прод без изменений; при приоритизации продукта — рассматривать усиление growth-элементов (счётчики, прогресс, простая геймификация) поверх текущей реферальной логики.

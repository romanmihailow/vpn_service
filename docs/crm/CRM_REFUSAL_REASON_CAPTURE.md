# CRM: сохранение причин отказа (опрос no handshake)

Реализовано сохранение и отчётность по ответам пользователей на опрос «Почему не стали пользоваться VPN?» (1–4) после отправки сообщения с опросом (no_handshake_survey).

---

## 1. Как ответ привязывается к подписке

- Обрабатываются только сообщения, похожие на ответ опроса: **одна цифра 1, 2, 3 или 4** (с опциональными пробелами/точкой в конце или эмодзи 1️⃣–4️⃣). Строки вроде `"12"`, `"123"` или `"1 2"` **не** считаются ответом.
- Подходящая подписка ищется через **`db.get_eligible_subscription_for_survey_answer(telegram_user_id)`**:
  - у пользователя есть подписка;
  - по этой подписке в `subscription_notifications` есть запись с `notification_type = 'no_handshake_survey'` (опрос уже отправлен);
  - по этой же подписке **нет** ни одной записи с типом `no_handshake_survey_answer_1` … `no_handshake_survey_answer_4` (ответ ещё не сохранён).
- Из всех таких подписок берётся одна: с самой поздней по `sent_at` записью опроса (`ORDER BY n_survey.sent_at DESC LIMIT 1`). Ответ сохраняется только по этой подписке.

В результате ответ «1»/«2»/«3»/«4» привязывается к последней подписке, по которой был отправлен опрос и ещё не сохранён ответ.

---

## 2. Как хранятся ответы

- Используется существующая таблица **`subscription_notifications`**.
- Тип уведомления для ответа:
  - **`no_handshake_survey_answer_1`** — не разобрался с настройкой  
  - **`no_handshake_survey_answer_2`** — пока не нужен  
  - **`no_handshake_survey_answer_3`** — пользуюсь другим VPN  
  - **`no_handshake_survey_answer_4`** — дорого  
- Вставка через **`db.create_subscription_notification(subscription_id, notification_type, telegram_user_id, expires_at)`**. Уникальность по `(subscription_id, notification_type)` не даёт сохранить два ответа по одной подписке.

---

## 3. Что отвечает бот пользователю

- **Новый ответ сохранён:**
  - для ответа **1** (не разобрался с настройкой):  
    `Спасибо за ответ! Это поможет нам улучшить сервис. Если захочешь, я могу помочь с подключением.`
  - для ответов **2, 3, 4**:  
    `Спасибо за ответ! Это поможет нам улучшить сервис.`
- **Ответ по этой подписке уже был сохранён** (повторное «1»/«2»/«3»/«4»):  
  `Ответ уже сохранён, спасибо.`
- Если подходящей подписки нет (опрос не отправлялся или ответ уже был), ответ опросу не считается, сообщение обрабатывается как обычный текст (AI-support и т.д.).

Тексты вынесены в `app/messages.py`:  
`SURVEY_ANSWER_CONFIRMATION`, `SURVEY_ANSWER_CONFIRMATION_SETUP_HELP`, `SURVEY_ANSWER_ALREADY_SAVED`.

---

## 4. Как отображаются причины в /crm_report

- В отчёте по-прежнему выводится строка:  
  **• опрос причины отказа: N** (число отправленных опросов за период).
- Ниже добавлен блок **«Причины отказа»** за тот же период (то же окно по `sent_at` в `subscription_notifications`):
  - не разобрался с настройкой: N  
  - пока не нужен: N  
  - пользуюсь другим VPN: N  
  - дорого: N  

Считаются уникальные `subscription_id` с типами `no_handshake_survey_answer_1` … `no_handshake_survey_answer_4` и `sent_at >= NOW() - (days)`, в `db.get_crm_funnel_report(days)` добавлены поля `no_handshake_survey_answer_1` … `no_handshake_survey_answer_4`.

# Аудит follow-up сообщений после первого handshake VPN

**Дата:** 2026-03-16  
**Цель:** выяснить, есть ли в боте короткий follow-up через ~30–60 секунд после первого успешного подключения, и построить таймлайн сообщений. Только анализ, код не менялся.

---

## 1. Jobs и задачи, связанные с handshake

### 1.1. `auto_new_handshake_admin_notification` (tg_bot_runner.py)

- **Интервал:** каждые **120 секунд** (`NEW_HANDSHAKE_ADMIN_INTERVAL_SEC = 120`).
- **Логика:** вызывает `wg.get_handshake_timestamps()`, получает подписки через `db.get_subscriptions_for_new_handshake_admin()` (активные подписки, по которым ещё не отправляли админу уведомление `new_handshake_admin`). Для подписок с handshake > 0:
  - **Пользователю** отправляется первое сообщение после handshake: `HANDSHAKE_USER_CONNECTED_TEXT` + `HANDSHAKE_USER_CONNECTED_KEYBOARD`, создаётся запись `notification_type='handshake_user_connected'`.
  - **Админу** отправляется сводка «Новых подписчиков с handshake».
- **Итог:** первое пользовательское сообщение «VPN подключён 👍» уходит при ближайшем прогоне job после появления handshake, т.е. в среднем через **0–120 сек** после фактического handshake (не фиксированные 30–60 сек).

### 1.2. `auto_handshake_followup_notifications` (tg_bot_runner.py)

- **Интервал цикла:** каждые **120 секунд** (`HANDSHAKE_FOLLOWUP_INTERVAL_SEC = 120`).
- **Источник кандидатов:** `db.get_handshake_followup_candidates(followup_type)` — подписки, у которых есть запись `handshake_user_connected` и с момента её `sent_at` прошёл интервал для данного типа.
- **Типы follow-up:**
  - **handshake_followup_10m** — через **10 минут** после `handshake_user_connected`.
  - **handshake_followup_2h** — через **2 часа** (только триал/промо).
  - **handshake_followup_24h** — через **24 часа** (только триал/промо).
  - **handshake_referral_nudge_3d** — через **3 дня** (только триал/промо), реферальный nudge.

### 1.3. Другие места с `wg.get_handshake_timestamps()` / handshake

- **config_check_now** (callback «Проверить подключение»): проверка handshake для показа «Да, всё работает» / «Нет, не получилось» — по действию пользователя, не по расписанию.
- **auto_config_checkpoint**: сообщение «Удалось подключиться к VPN?» через ~3 мин после **отправки конфига** (по `config_checkpoint_pending`), не привязано к handshake по времени.
- **support/context_builder.py**, **support/actions.py**: использование handshake для контекста поддержки.
- **auto_no_handshake_reminder**: напоминания тем, у кого **нет** handshake (2h, 24h, 5d после выдачи конфига).

Отдельного job’а, который отправляет что-то пользователю **строго через 30–90 сек** после handshake, нет.

## 2. Таймлайн сообщений после первого handshake

Отсчёт от момента **фактического handshake** (peer в WG отправил handshake).

| Момент | Когда | Сообщение пользователю | Кнопки |
|--------|--------|------------------------|--------|
| **T+0 … T+2 min** | При следующем прогоне `auto_new_handshake_admin_notification` (раз в 2 мин) | `HANDSHAKE_USER_CONNECTED_TEXT`: «VPN подключён 👍», про 7 дней триала, тариф 3 мес / 270 ₽ | 💎 Закрепить доступ — 270 ₽ (`pay:open`), 📅 Все тарифы (`pay:open`) |
| **T+10 min** | Ровно через 10 мин после **отправки** первого сообщения (`handshake_user_connected.sent_at`) | `HANDSHAKE_FOLLOWUP_10M_TEXT`: «VPN подключён 👍 Всё открывается нормально?» | Всё работает (`vpn_ok:{sub_id}`), 🧑‍💻 Нужна помощь (url) |
| **T+2 h** | Через 2 ч после первого сообщения (только триал/промо) | `HANDSHAKE_FOLLOWUP_2H_TEXT`: закрепить доступ, 3 мес 270 ₽, /buy | 🛒 Закрепить доступ, 🤝 Пригласить друга, 🧑‍💻 Нужна помощь |
| **T+24 h** | Через 24 ч после первого сообщения (только триал/промо) | `HANDSHAKE_FOLLOWUP_24H_TEXT`: закрепить доступ, 3 мес 270 ₽ | Нет inline-кнопок |
| **T+3 days** | Через 3 дня после первого сообщения (только триал/промо) | `HANDSHAKE_REFERRAL_NUDGE_3D_TEXT`: реферальная программа, /ref | 🤝 Пригласить друга |

Дополнительно: при нажатии «Всё работает» в 10m пользователь получает `VPN_OK_ANSWER_TEXT` (upsell + реферал) и кнопки «🛒 Купить подписку», «🤝 Пригласить друга».

## 3. Наличие короткого follow-up (30–90 сек)

**Ответ: Short confirmation follow-up is missing.**

Сообщения пользователю после handshake:

- Первое — при прогоне job раз в **2 минуты**, т.е. доставка в интервале **0–120 сек** после handshake, а не в фиксированном окне 30–90 сек.
- Отдельного сообщения «всё ок, подключились» через 30–60 или 30–90 сек **нет**.

То есть выделенного короткого confirmation follow-up через ~30–60 сек в боте нет.

## 4. Дублирование продаж по follow-up

Какие follow-up содержат upsell (тариф, оплата, закрепить доступ):

| Follow-up | Upsell в тексте | Кнопки покупки/тарифов |
|-----------|------------------|-------------------------|
| **T+0…2 min** (первое сообщение) | Да (3 мес 270 ₽) | Да: «Закрепить доступ», «Все тарифы» |
| **10m** | Нет (только «всё открывается нормально?») | Нет в самом сообщении; после «Всё работает» — текст с upsell + «Купить подписку», «Пригласить друга» |
| **2h** | Да (закрепить доступ, 3 мес 270 ₽, /buy) | Да: «Закрепить доступ», «Пригласить друга», «Нужна помощь» |
| **24h** | Да (закрепить доступ, 3 мес 270 ₽, /buy) | Нет (только текст) |

Итого: upsell есть в первом сообщении (T+0…2 min), в 2h и 24h; в 10m — косвенно (через ответ на «Всё работает»).

## 5. Краткие ответы

1. **Есть ли short confirmation follow-up?**  
   **Нет.** Отдельного сообщения через ~30–60 сек после handshake нет. Первое сообщение уходит при следующем прогоне job (0–120 сек).

2. **Есть ли окно между handshake и 10m?**  
   **Да.** Между первым сообщением (до 2 мин после handshake) и 10m — около **8+ минут** без новых сообщений (если считать от первого сообщения — ровно 10 мин до 10m).

3. **Можно ли безопасно добавить сообщение через ~60 сек?**  
   **Да.** Можно ввести новый тип уведомления (например, `handshake_short_confirmation`) и job/логику «отправить через 60 сек после `handshake_user_connected.sent_at`». Условия: не дублировать первое сообщение (отправлять только «всё ок» / короткий confirmation), не менять условия для 10m/2h/24h (они завязаны на `handshake_user_connected` и его `sent_at`).

4. **Повлияет ли это на существующие jobs?**  
   **Нет**, если только добавить новый тип уведомления и отдельную отправку. Логика `get_handshake_followup_candidates` для 10m/2h/24h опирается на наличие `handshake_user_connected` и интервалы от его `sent_at`; добавление ещё одного сообщения через ~60 сек от того же `sent_at` не меняет эти интервалы и не требует менять существующие job’ы.
